# Deployment Guide
**Operaxon OS — Phase 3**

How we deploy customer instances. From config to live.

---

## Architecture

Each customer = one isolated tenant:

```
operaxon.io (gateway router)
  ├── tenant_abc123 → operaxon-abc123.fly.dev:4001
  ├── tenant_def456 → operaxon-def456.fly.dev:4002
  └── tenant_ghi789 → operaxon-ghi789.fly.dev:4003
```

No data ever crosses tenant boundaries.

---

## Deployment Targets

### Fly.io (Recommended)

Fast, global, managed. Default for all tiers.

```bash
cd deployments/{tenantId}
flyctl apps create {instance-name}
flyctl secrets set $(cat .env | grep -v '#' | xargs)
flyctl deploy
```

**Auto-configured via `fly.toml`** (generated by Provisioner).

### Docker Compose (Self-Hosted / Enterprise)

```bash
cd deployments/{tenantId}
docker-compose up -d

# Check health
curl http://localhost:{port}/health
```

### Railway

```bash
cd deployments/{tenantId}
railway up
```

## Tenant Directory Structure

Every provisioned customer gets:

```
deployments/
  {tenantId}/
    .env                    ← NEVER commit — contains API keys
    operaxon.config.json    ← Full runtime config
    fly.toml                ← Fly.io config (if fly.io target)
    docker-compose.yml      ← Docker config
    deploy-manifest.json    ← Deployment spec
    agents/
      {agent-name}.json     ← Per-agent config

data/
  {tenantId}/               ← Customer data (isolated)
    meridian/               ← MERIDIAN memory
    logs/                   ← Runtime logs
    audit/                  ← Audit trail
    usage.json              ← Usage counters
    governor-state.json     ← GOVERNOR state
```

## Health Checks

All instances expose `GET /health`. Must return 200 to be marked live.

```json
{
  "status": "ok",
  "timestamp": "2026-03-31T16:00:00Z",
  "tenantId": "tenant_abc123"
}
```

Health checker polls every 30 seconds. 3 consecutive failures → `degraded`.

---

## Rollback

If a deployment fails:

1. **Fly.io:** `flyctl releases rollback --app {instance-name}`
2. **Docker:** `docker-compose down` → redeploy with previous config
3. **Config:** Snapshot restored from `deployments/snapshots/{tenantId}/pre-deploy/`

Rollback is automatic on deploy failure. Manual rollback:

```bash
# After manual fix:
node -e "const { RollbackManager } = require('@operaxon/deployment'); ..."
```

## Adding a New Customer (Full Flow)

```bash
# 1. Customer fills questionnaire → saved as JSON
# 2. Run provisioner
npx ts-node -e "
import { Provisioner } from '@operaxon/onboarding';
const p = new Provisioner({ outputBaseDir: './deployments', basePort: 4001 });
const config = p.provision(questionnaire, tenantId);
console.log('Config ready:', config.outputDir);
"

# 3. Deploy
cd deployments/{tenantId}
flyctl deploy

# 4. Verify health
curl https://{instance}.fly.dev/health

# 5. Webhook fires → customer notified
```

Total time: **~5 minutes** from questionnaire to live.

## Monitoring

- Health checks run every 30s (auto-managed)
- Failed health checks trigger PagerDuty/webhook alert
- `GET /dashboard/status` — per-tenant status
- Logs: `data/{tenantId}/logs/`
- Audit: `data/{tenantId}/audit/audit-{date}.jsonl`

---

## Port Allocation

Ports are auto-allocated starting at 4000.

| Port | Tenant |
|---|---|
| 4000 | Reserved (internal) |
| 4001 | First customer |
| 4002 | Second customer |
| ... | ... |

On Fly.io, each customer gets their own subdomain — port doesn't matter externally.
